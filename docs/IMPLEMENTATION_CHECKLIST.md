# SKG: Scorekeeper Implementation Checklist

> 這份清單是 GDD 與 prototype 之間的修正主線。每次修改前後都要檢查它，完成一項就更新狀態與備註。
>
> 狀態：`[ ]` 待處理、`[~]` 進行中、`[x]` 已完成、`[!]` 被阻塞或需要重新設計

## 工作流規則

- 一次只處理一個主要修正，避免同時改散多條主線。
- 每次開始修改前，先檢查本清單，確認目前要處理的項目與驗收條件。
- 每次修改完成後，必須重新檢視本清單，更新完成狀態與實際結果。
- 每次修改後至少執行 `npm run lint` 與 `npm run build`。
- 能在瀏覽器驗證的流程，要實際走過並記錄結果；不能只看程式碼猜測。
- 如果發現修改影響其他清單項目，要在「備註／驗證紀錄」留下註記。
- 沒有完成的項目不能標記為完成；若只完成一部分，使用 `[~]` 並寫清楚剩餘差距。
- 每次完成一個工作單位後，必須 `git add`、`git commit`、`git push` 到 GitHub，不能只保留本機修改。

## 已確定的介面方向：玩家模式與 Debug 工具

### A. 將左側 SKG／Evidence 面板改成 Developer Debug Mode `[~]`

- [x] 一般玩家模式預設完全隱藏左側面板，手機成為畫面唯一的主要互動焦點。
- [x] 保留現有左側面板程式碼，不直接刪除，改為開發者專用工具。
- [x] 支援 `Ctrl + Shift + D` 切換 Debug 面板。
- [x] 支援網址參數 `?debug=true` 直接開啟 Debug Mode。
- [x] Debug Mode 可查看重要進度旗標與線索解鎖狀態。
- [x] Debug Mode 提供 GDD 謎題 1–10 的統一章節快照與對應 App 跳轉。
- [ ] 後續逐步加入快速跳到遊戲內 37／184／256 分數節點，以及測試三個結局的功能。
- [x] Debug 面板的文字或狀態不可在一般玩家模式洩漏答案。
- [~] 玩家模式隱藏面板後，手機已在現有直向版面置中；橫向手機與不同 viewport 的最終尺寸需等待 Claude 的 layout 分支整合後複驗。

**驗收：** 一般網址開啟時看不到左側面板；按下 `Ctrl + Shift + D` 或使用 `?debug=true` 時可顯示，再次按快捷鍵可隱藏，且不影響遊戲進度。

### B. 在手機內加入引導式線索 App `[ ]`

- [ ] 將玩家可用的線索功能放回手機世界內，而不是放在外部系統面板。
- [ ] App 顯示玩家已經發現的線索摘要。
- [ ] App 可顯示尚待追查的方向，但只提供引導，不直接給搜尋詞、密碼、高度序列或謎底。
- [ ] 未發現的線索應以疑問、模糊描述或待調查狀態呈現。
- [ ] 新提示必須由實際遊戲進度與玩家已取得的證據解鎖。
- [ ] App 的名稱與世界觀整合後再決定，可考慮 `Recovered Files`、`Notes` 或 `Investigation`。
- [ ] 不重複其他 App 已完整呈現的內容，只保存可幫助玩家回想與繼續追查的摘要。

**驗收：** 遇到困難的玩家可以從 App 知道下一個調查方向，但不能只靠這個 App 直接得到任何主要謎題答案。

## 第一階段：修正主線節奏（最高優先）

### 1. 讓第 37 分固定成為必死關卡 `[ ]`

- [ ] 前 36 個障礙不應因隨機因素讓玩家過早死亡。
- [ ] 第 37 段固定出現「看似可通過、實際必撞」的幽靈牆／障礙。
- [ ] 玩家前幾次遊玩都能穩定得到 37 分死亡的體驗。
- [ ] 死亡原因與 UI 明確指向第 37 段，而不是普通隨機碰撞。

**驗收：** 連續測試數次，確認主線觸發的是 `Gate 37`，而不是在 37 前隨機死亡。

### 2. 延後排行榜的 `-65535` 真相 `[ ]`

- [ ] 前期排行榜只顯示大量 37、少量 38–62、ARC_184 的 184，以及 `65,535 ENTRIES HIDDEN`。
- [ ] 玩家突破 37 前，不顯示 Noah 的 `-65535`。
- [ ] 玩家超過 ARC_184 的 184 後，才出現 `SORT ORDER INVALID`、`SIGNED VALUE DETECTED` 等提示。
- [ ] 玩家進入隱藏排行榜後，才可找到 `NOAH_KADE — -65535`。
- [ ] 溢位原因在正確的劇情節點揭露，而不是序章直接說明。

**驗收：** 新遊戲第一次死亡與前期調查看不到 `-65535`；完成隱藏路線並超過 184 後才看到它。

### 3. 對齊前三次死亡與序章節奏 `[ ]`

- [ ] 第一次死亡：37 分，讓玩家感到困惑。
- [ ] 第二次死亡：37 分，開始確認這不是偶然。
- [ ] 第三次死亡後：排行榜自動展開並正式引導調查 ARC_184。
- [ ] 不應在第二次死亡就提前展示完整終局真相。

**驗收：** 重置遊戲後完整走過前三次死亡，確認提示順序與 GDD 一致。

## 第二階段：鎖住真正的謎題依賴鏈

### 4. 建立不可跳過的調查主線 `[~]`

- [~] 37 分死亡／排行榜後才允許 ViewTube 的 ARC_184 搜尋；前三次死亡節奏仍待第一階段修正。
- [x] 搜尋並觀看 ARC_184 影片後，才允許使用 Lumen Arc 關鍵詞進入 AmazeMart 後續。
- [ ] 找到舊版檔案／Archive 線索後，才引導 AmazeMart。
- [ ] 購買 Lumen Arc 後，才解鎖截圖資料夾。
- [ ] 閱讀截圖後，才得到 `SKG: Skyline 256` 與 SilverKite_Games。
- [x] 取得原始標題線索後，才允許 Wayback Browser 開啟 2014 Silver Kite Games 歷史頁。
- [x] 追查 Browser 的 Silver Kite Games 舊網站後，才允許 FaceSpace 搜尋 Noah。
- [ ] FaceSpace 搜尋 Noah 並查看最早貼文／Q&A 後，才得到 `184-37-256`。
- [x] Messages 密碼登入已使用 `unlockedAdminLogin` 作為前置門檻；過早猜中會顯示幽默錯誤。
- [ ] 登入舊帳號後，才解鎖高度序列。
- [~] ViewTube、AmazeMart、Browser、FaceSpace 與 Messages 的主要猜答案入口已設門檻；其餘內容頁與完整 1–10 自然推進仍待後續逐章串接。

**驗收：** 新遊戲不能直接從 Messages 或任何單一 app 取得完整答案；每個關鍵答案都要有前置條件。

### 5. 讓應用程式內容成為謎題，而非直接答案頁 `[~]`

- [~] 已阻擋已知關鍵詞與密碼的提前使用；其他過早出現的完整答案仍待逐頁盤點。
- [ ] 保留必要的模糊提示、矛盾與交叉證據。
- [ ] 讓玩家必須在至少兩個 app 之間比對資料，才能形成下一步推論。
- [~] `unlockedAdminLogin`、`discoveredOriginalTitle`、`discoveredSKGHistory`、`watchedVideo` 等狀態已成為部分入口的實際條件；`archiveDownloaded` 等其餘旗標仍待串接。

**驗收：** 逐一檢查每個主要狀態是否真的會改變可互動內容，而不只是記錄旗標。

## 第三階段：補足重要互動與高潮

### 6. 完成截圖／紙張謎題互動 `[ ]`

- [ ] 保留目前的放大閱讀功能。
- [ ] 加入旋轉或排列操作，至少有一個可驗證的排列目標。
- [ ] 錯誤排列不應直接解鎖完整答案。
- [ ] 正確排列後才解鎖 `SKG: Skyline 256` 與開發者帳號線索。

**驗收：** 玩家必須完成互動才能取得截圖謎題的關鍵線索。

### 7. 讓隱藏高度路線成為玩家驗證的 mechanic `[ ]`

- [ ] 高度序列仍為 `184, 172, 149, 133, 121, 118, 126, 143`。
- [ ] 路線必須在前置調查完成後才可使用。
- [ ] 高度判定要穩定、可理解、可重試。
- [ ] debug 面板不能取代故事中的推理過程。
- [ ] 突破第 37 段後，畫面與內容逐層轉為原始／像素／開發殘骸風格。

**驗收：** 玩家能靠已取得的證據穩定突破 37，且錯誤輸入會有清楚回饋。

### 8. 完整呈現 256 終點與分數溢位高潮 `[ ]`

- [ ] 第 256 段出現 `SERVICE TERMINATED` 的垂直黑牆。
- [ ] 玩家必須使用高度 `0`，完全不點擊，從底部穿過。
- [ ] 完成時顯示 `SKYLINE COMPLETE` 與 `THANK YOU FOR PLAYING`。
- [ ] 分數依序跳動：`256 → 65535 → -65535`。
- [ ] 完成後才進入 Credit 與三選一結局。

**驗收：** 玩家能明確感受到「抵達真正終點」，而不是只看到一般的完成 overlay。

- [ ] 不要在過早階段直接把主角身份寫成 Noah。
- [ ] 保留 Mara 記憶衰退與零碎提示的效果。

- [ ] 讓親子關係的揭露發生在調查累積之後。

**驗收：** 新玩家在前期不知道完整親子真相，並能在中後段感受到揭露的情緒重量。

### 10. 對齊三個結局的敘事後果 `[ ]`

- [ ] 提交分數：成為第一名，但 Noah 的負分仍被忽略。
- [ ] 公開秘密：大家知道真相，但 SKG 移除舊版／關閉資料，最後什麼也沒留下。

- [ ] 三個結局都要有不同且完整的後果文字，而不只是標題變化。

**驗收：** 三個選項各自重跑一次，確認後果與 GDD 第 10 節一致。
